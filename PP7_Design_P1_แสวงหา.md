# กระบวนการ P1: การแสวงหา_ผู้สมัครงาน (LDC-PAO-PM-001)
**สถานะ:** ยืนยันแบบร่างแล้ว (Confirmed)
**วันที่:** 2026-06-06

## 1. กระบวนการเดิม
ผู้สมัครสแกน QR Code → กรอกฟอร์มออนไลน์ → ข้อมูลถูกบันทึกแยกส่วน (เช่น Google Sheet หรืออีเมล) → HR ต้องดึงข้อมูลมาคัดกรองและตามหาไฟล์ประวัติด้วยตนเอง

## 2. จุดที่ AI เข้ามาช่วย
- ตรวจสอบความครบถ้วนของฟิลด์ข้อมูล (Validation) ก่อนบันทึก
- สกัดข้อมูลสำคัญจากไฟล์ Resume/CV ที่แนบมาอัตโนมัติ (OCR/Parser)
- จัดหมวดหมู่ผู้สมัครเบื้องต้นตามตำแหน่งงานที่เลือก

## 3. กระบวนการใหม่
ผู้สมัครสแกน QR Code → กรอกฟอร์มที่เชื่อมตรงกับ Web PP7 → ระบบบันทึกข้อมูลลง Database ของ Web PP7 พร้อมสถานะ "ผู้สมัครใหม่" → ระบบแจ้งเตือน HR ผ่าน Telegram ทันที

## 4. ความซ้ำซ้อน/ความซับซ้อน
- การเก็บข้อมูลซ้ำซ้อนในหลายที่ (Sheet + Email)
- การตรวจสอบ PDPA ด้วยความจำหรือเอกสารกระดาษ
- การตามหาประวัติผู้สมัครเมื่อเวลาผ่านไป
- การอัปเดตตำแหน่งงานว่างที่ไม่เป็นปัจจุบัน

## 5. กระบวนการใหม่หลังลดความซ้ำซ้อน
- Single Source of Truth บน Web PP7 เท่านั้น
- มีตารางจัดการตำแหน่งงานว่าง (Position List) ที่ Admin อัปเดตได้ง่าย
- สถานะผู้สมัครอัปเดตแบบ Real-time
- ไฟล์แนบและข้อมูล PDPA ถูกจัดเก็บเป็นระบบและเชื่อมโยงกับโปรไฟล์ผู้สมัครโดยตรง

## 6. Task หลังลดความซ้ำซ้อน
- **HR/ผู้คัดกรอง**: ตรวจสอบ Dashboard "ผู้สมัครใหม่" ใน Web PP7 → กด "คัดกรองเบื้องต้น" → หากผ่าน เปลี่ยนสถานะเป็น "รอประเมิน (P2)" โดยอัตโนมัติ
- **Admin/ระบบ**: อัปเดตรายการตำแหน่งงานว่าง (Position List) ในระบบเมื่อมีความต้องการรับสมัคร

## 7. แนวทางออกแบบโครงสร้าง Web PP7 (Data Structure)
### Table: `positions` (จัดการตำแหน่งงาน)
- `id` (UUID)
- `title` (String) - ชื่อตำแหน่งงาน
- `department` (String) - แผนก
- `is_vacant` (Boolean) - สถานะเปิดรับ (True/False)
- `updated_at` (Timestamp)

### Table: `candidates` (ข้อมูลหลักผู้สมัคร)
- `id` (UUID)
- `position_id` (UUID, FK) - เชื่อมกับตาราง positions
- `full_name` (String)
- `contact_info` (JSON) - เบอร์โทร, อีเมล, ที่อยู่
- `resume_url` (String) - ลิงก์ไฟล์ที่อัปโหลด
- `pdpa_consent` (Boolean) - ยินยอม PDPA (ต้องเป็น True ถึงจะบันทึกได้)
- `pdpa_consent_date` (Timestamp) - เวลาที่กดยินยอม
- `status` (Enum) - 'New', 'Screening', 'Passed_to_P2', 'Rejected'
- `created_at` (Timestamp)

### Table: `candidate_references` (ข้อมูลบุคคลที่สาม)
- `id` (UUID)
- `candidate_id` (UUID, FK)
- `relation` (String) - ความสัมพันธ์ (เช่น บุคคลอ้างอิง, กรณีฉุกเฉิน)
- `name` (String)
- `phone` (String)

## 8. ความเชื่อมโยงของระบบ (Step 3: Data Flow)
- **ทิศทางการไหลของข้อมูล:** P1 (แสวงหา) -> ส่งต่อข้อมูลทั้งหมด -> P2 (หยั่งประเมิน) -> คัดเลือก -> P3 (จับคู่คนกับงาน)
- **Trigger การส่งต่อ:** เมื่อ HR/ผู้คัดกรอง ใน P1 เปลี่ยนสถานะผู้สมัครจาก `Screening` เป็น `Passed_to_P2`
- **ข้อมูลที่ถูกส่งต่อ (Data Payload):** 
  1. ข้อมูลส่วนตัวและประวัติ (ชื่อ, ตำแหน่งที่สมัคร, Resume URL, ข้อมูลติดต่อ)
  2. สถานะความยินยอม PDPA (ต้องเป็น True)
  3. หมายเหตุ/ผลการคัดกรองเบื้องต้นจาก P1 (ถ้ามี)
- **การแสดงผลในฝั่ง P2:** Dashboard 'รอประเมิน' จะดึงข้อมูลจากตาราง `candidates` โดยตรง (ไม่มีการก๊อปปี้ข้อมูลซ้ำ) แสดงแบบ Read-Only พร้อมปุ่ม 'เริ่มประเมิน'
- **ข้อดีของ Flow นี้:** ผู้สมัครไม่ต้องกรอกข้อมูลซ้ำ (Zero Re-entry), ข้อมูลเป็น Real-time จากแหล่งเดียว (Single Source of Truth), ลดความผิดพลาดจากการคัดลอกข้อมูลระหว่างแผนก
